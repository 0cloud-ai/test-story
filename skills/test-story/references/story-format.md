# 测试故事 Markdown 格式规范

## 文件结构

一个测试故事由以下部分组成：

```markdown
# 故事标题

\```meta
target: http://localhost:8080
scene: api
\```

## 楔子（可选，跳过不执行）

背景叙述...

---

## 第一章 章节名

测试段落...

---

## 第N章 章节名

测试段落...

---

## 尾声（可选，跳过不执行）

总结叙述...
```

## 各元素说明

### H1 标题
故事标题，同时作为测试报告的名称。每个文件只有一个 H1。

### Meta 块
用三个反引号包裹，语言标记为 `meta`：

| 字段 | 必填 | 说明 |
|------|------|------|
| `target` | 是 | 被测 API 的 base URL，如 `http://localhost:8080` |
| `scene` | 否 | 测试场景类型，默认 `api`。预留字段，目前主要使用 `api` |

### H2 章节
每个 H2 标题开启一个测试章节，代表一个测试阶段（如注册、搜索、下单等）。

**特殊章节**（不执行测试）：
- **楔子**：故事开头的背景介绍，用于建立上下文
- **尾声**：故事结尾的总结，用于收束叙事

### 段落
章节内的每个段落描述一个或多个测试动作。段落之间用空行分隔。

### 水平线 `---`
纯装饰性分隔符，不影响测试逻辑。

## 写作指南

### 1. 角色设定
故事至少有一个"用户角色"走正常流程，可以有一个"测试角色"执行异常和边缘测试：
- **用户角色**（如"叶知秋"）：走完正常业务流程
- **测试角色**（如"陆青"）：验证异常场景、安全边界

### 2. 嵌入测试数据
在叙事中自然地包含具体数据值：
- 用户名、邮箱、密码等注册信息
- 搜索关键词
- 商品名称、价格、数量
- 地址、手机号等表单数据

示例：
> 她给自己取的用户名就是本名"叶知秋"，邮箱用的是 yezhiqiu@mail.com，密码是她的老习惯——Qiuye#1998

### 3. 描述预期行为
在叙事中明确描述系统的响应：

**成功场景**：
> 注册很顺利。系统认识她了，给她的身份标记是"普通读者"。

**字段校验**：
> 她确认了一下自己的个人信息都保存正确了，唯独密码没有被显示出来——这是应该的。

**数量和计算**：
> 总共5条结果。
> 小计是73.60元，等于单价乘以数量，计算无误。

**格式校验**：
> 订单号是"ORD-"开头的一串编码，格式规整。

### 4. 异常和安全测试
通过"测试角色"的行为描述边缘情况：

**重复操作**：
> 他用同样的用户名和邮箱又注册了一次，系统识别出这是重复注册，拒绝了

**缺失字段**：
> 他试了一次不填邮箱的注册，系统同样拒绝了，明确指出邮箱是必填的

**越权访问**：
> 他用另一个已登录的测试账号去查看叶知秋的订单，系统同样拒绝了，理由是无权访问该资源

**幂等性**：
> 他模拟支付成功的通知因为网络抖动被重复发送了一遍。系统正常接受了第二次通知，但订单状态没有再变化

**极端输入**：
> 把分页的页码设到99999去搜"三体"。结果返回的是空列表，但总数仍然正确地显示5条

### 5. 语言
故事可以用任何语言书写，中文是默认惯例。叙事风格应自然流畅，像讲故事一样，避免直接写技术指令。

## 最小模板

```markdown
# [故事标题]

\```meta
target: http://localhost:8080
scene: api
\```

## 第一章 [章节名]

[角色名]做了什么操作，用了什么数据。系统返回了什么结果，符合预期。

[测试角色]试了一个异常场景。系统正确地拒绝了，给出了合理的错误提示。
```
